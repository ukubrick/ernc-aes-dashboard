## 1. Alcance y aclaración sobre “plan”

En los documentos revisados no aparece un campo llamado **plan** asociado a cada endpoint. Lo que sí aparece es la **API o servicio base** al que pertenece cada endpoint y su **tag/categoría**. En este resumen se usa la columna **Plan/API** para indicar si el endpoint pertenece a **SIP Sistema de Información Pública** o a **Operaciones**.

Tampoco aparece un endpoint o tag explícito llamado **Adquisición** o **Adquisiciones**. Por lo tanto, la expresión “peticiones de adquisición” se interpreta como **peticiones para adquirir/consultar datos desde la API**.

## 3. Elementos necesarios para realizar peticiones

### 3.1. URL base

- **SIP:** `https://sipub.api.coordinador.cl`
- **Operaciones:** `https://operacion.api.coordinador.cl`

### 3.2. Autenticación

Ambas especificaciones declaran autenticación tipo **API key** mediante parámetro de consulta:

```text
user_key=TU_API_KEY
```

La API key se agrega al final de la URL, por ejemplo:

```text
https://sipub.api.coordinador.cl/generacion-programada-pcp/v4/findByDate?startDate=2026-06-22&endDate=2026-06-22&page=0&limit=100&user_key=TU_API_KEY
```

### 3.3. Método HTTP

Los endpoints documentados en estos archivos son principalmente de tipo **GET**. La consulta se realiza enviando parámetros en la URL, no en el cuerpo de la petición.

### 3.4. Parámetros comunes

| Parámetro | Uso típico |
|---|---|
| `user_key` | API key asignada. Obligatoria para la autenticación. |
| `startDate` | Fecha inicial. Normalmente formato `YYYY-MM-DD`. |
| `endDate` | Fecha final. Normalmente formato `YYYY-MM-DD`. |
| `date` | Fecha única de consulta. Formato habitual `YYYY-MM-DD`. |
| `page` | Número de página. Generalmente inicia en `0`. |
| `limit` | Tamaño de página o cantidad de registros por página. |
| `id` | Identificador del recurso a consultar. Puede ir como query o path. |
| `search` | Búsqueda textual general en endpoints que la permiten. |
| `ordering` | Campo de ordenamiento en endpoints que lo permiten. |

### 3.5. Códigos de respuesta comunes

| Código | Significado práctico |
|---:|---|
| `200` | Consulta exitosa. |
| `400` | Parámetros incorrectos o mal formateados. |
| `401` | No se envió API key. |
| `403` | API key incorrecta o sin permisos. |
| `404` | Endpoint o recurso no encontrado. |
| `500` | Error interno del servicio. |

### 3.6. Paginación

Muchos endpoints devuelven respuestas paginadas. Dependiendo del servicio, el contenido puede venir en campos como `data` o como `content`, junto con metadatos como `page`, `pageSize`, `totalElements` y `totalPages`. Para adquirir todo el dataset, se debe iterar desde `page=0` hasta completar las páginas disponibles.

## 5. Ejemplo Python reutilizable

```python
import requests
import pandas as pd

API_KEY = "TU_API_KEY"

def consultar_api(base_url, endpoint, params=None, timeout=60):
    params = dict(params or {})
    params["user_key"] = API_KEY

    url = base_url.rstrip("/") + endpoint
    response = requests.get(url, params=params, timeout=timeout)

    if response.status_code == 401:
        raise RuntimeError("No se envió API key o el parámetro user_key está ausente.")
    if response.status_code == 403:
        raise RuntimeError("API key incorrecta o sin permisos para este endpoint.")
    if response.status_code == 400:
        raise RuntimeError(f"Parámetros inválidos: {response.text}")
    if response.status_code == 404:
        raise RuntimeError(f"Recurso no encontrado: {url}")
    response.raise_for_status()

    return response.json()

# Ejemplo: adquisición de generación programada PCP desde SIP
data = consultar_api(
    base_url="https://sipub.api.coordinador.cl",
    endpoint="/generacion-programada-pcp/v4/findByDate",
    params={
        "startDate": "2026-06-22",
        "endDate": "2026-06-22",
        "page": 0,
        "limit": 100
    }
)

# Normalización básica: algunas APIs responden con "data", otras con "content"
registros = data.get("data", data.get("content", []))
df = pd.json_normalize(registros)
print(df.head())
```

## 6. Función para adquirir todas las páginas

```python
import requests
import pandas as pd

API_KEY = "TU_API_KEY"

def adquirir_todas_las_paginas(base_url, endpoint, params=None, limit=100, max_pages=10000):
    registros = []
    page = 0
    params_base = dict(params or {})
    params_base["limit"] = limit
    params_base["user_key"] = API_KEY

    while page < max_pages:
        params_page = dict(params_base)
        params_page["page"] = page

        response = requests.get(base_url.rstrip("/") + endpoint, params=params_page, timeout=60)
        response.raise_for_status()
        payload = response.json()

        lote = payload.get("data", payload.get("content", []))
        if not lote:
            break

        registros.extend(lote)

        total_pages = payload.get("totalPages")
        if total_pages is not None and page >= int(total_pages) - 1:
            break

        page += 1

    return pd.json_normalize(registros)

df = adquirir_todas_las_paginas(
    base_url="https://sipub.api.coordinador.cl",
    endpoint="/costo-marginal-online/v4/findByDate",
    params={
        "startDate": "2026-06-22",
        "endDate": "2026-06-22"
    },
    limit=100
)

df.to_csv("costo_marginal_online.csv", index=False, encoding="utf-8-sig")
```

## 9. Recomendación práctica de uso

Para adquisición periódica de datos, conviene crear un script por familia de endpoints. Por ejemplo, un módulo para `generación`, otro para `demanda`, otro para `costo marginal`, otro para `SSCC` y otro para `maestros/topología`. En cada ejecución se debe registrar: endpoint consultado, parámetros usados, fecha/hora de consulta, estado HTTP, cantidad de registros y respuesta cruda o normalizada.

Para procesos diarios, se recomienda guardar primero el JSON bruto y después normalizar a tablas. Esto permite re-procesar los datos si cambia el modelo de datos o si se requiere auditoría.
